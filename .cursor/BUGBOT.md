# Cursor Bugbot review guide

Review this repository as a first-playable browser game prototype built with Next.js,
React, TypeScript, and Phaser. Prioritize defects that can break the playable game,
deployment, or future iteration velocity. Avoid blocking on subjective art direction,
content polish, or prototype scope unless a change introduces a concrete regression.

## Project shape

- `src/app/page.tsx` renders the full-screen game host.
- `src/game/GameCanvas.tsx` is a client component that dynamically imports Phaser and
  owns the Phaser game lifecycle.
- `src/game/scenes/DungeonScene.ts` contains most runtime gameplay: scene preload,
  animation setup, input, collision, enemies, projectiles, pickups, HUD, audio, and
  game-over flow.
- `src/game/maps/startingDungeon.ts` generates the room-and-corridor dungeon and tile
  semantics.
- `src/game/assets/manifest.ts` is the source of truth for asset keys, public paths,
  frame sizes, directions, and audio assets.
- `public/assets/**` contains the generated spritesheets, sprite metadata, tiles,
  effects, UI, and audio that Phaser loads at runtime.
- Asset generation and processing utilities live under `tools/` and `scripts/`.

## High-priority review areas

1. **Next.js client/server boundaries**
   - Keep Phaser, `window`, `document`, canvas, pointer, and audio APIs inside client
     components or dynamically imported game modules.
   - Watch for changes that make the app fail during SSR, static rendering, or
     `next build`.
   - If metadata changes reference OpenGraph or Twitter image assets, verify the
     referenced files exist under `public/` or are generated by the route being used.

2. **Phaser lifecycle and cleanup**
   - `GameCanvas` should create only one game instance per host and destroy it on
     unmount.
   - Scene changes should clean up timers, event listeners, tweens, input handlers,
     and pooled objects that can survive restart/game-over cycles.
   - Treat leaked callbacks or duplicated scene state after restart as blocking bugs.

3. **Gameplay correctness**
   - Movement, aiming, projectile cooldowns, ammo counts, seeker targeting, enemy
     respawns, heart pickups, power-ups, score, game-over, and restart logic should
     preserve their invariants across repeated runs.
   - Collision changes must keep playable tiles, low walls, chasms, bridges, props,
     enemies, pickups, and projectiles consistent with `startingDungeon.ts`.
   - Difficulty scaling should not create unavoidable instant damage, spawn enemies on
     the player, exceed intended caps, or soft-lock a run.

4. **Asset manifest consistency**
   - New or changed assets should update `assetManifest`, README asset references, and
     sprite metadata together.
   - Check frame dimensions, row/column assumptions, animation keys, and directional
     ordering against the generated JSON files.
   - Public asset paths must stay absolute from `public/` (for example,
     `/assets/effects/staff-bolt-sprite-sheet.png`) so Phaser can load them in a
     deployed Next app.

5. **Input, accessibility, and browser behavior**
   - Keyboard controls, pointer aim/fire, debug toggle, and audio mute controls should
     remain usable after focus changes and restarts.
   - The page intentionally fills the viewport and hides overflow; flag changes that
     introduce unexpected scrolling, canvas stretching, or broken resize behavior.
   - UI shown through Phaser should still expose clear keyboard/mouse affordances in
     the game instructions or overlays when controls change.

6. **Build and type health**
   - Prefer `npm run build` as the integration check for Next.js/Turbopack behavior.
   - Prefer `npx tsc --noEmit --incremental false` for a side-effect-free TypeScript
     check.
   - Do not rely on `npm run lint` until the script is migrated away from `next lint`;
     with current dependencies, that script treats `lint` as a project directory.

## What to avoid blocking on

- Do not require hand-polished art, complete game balance, or final accessibility
  treatment for this prototype unless the diff makes the current playable experience
  objectively worse.
- Do not request broad decomposition of `DungeonScene.ts` unless a diff makes a real
  bug likely or unreviewable; targeted extraction is useful only when it protects the
  behavior being changed.
- Do not flag generated asset files solely because they are large or binary-like if
  their manifest references and metadata remain consistent.
